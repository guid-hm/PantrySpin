Baseline2 hr review

Where the prototype actually stands.

Before estimating, be honest about what you have. The PantrySpin prototype is a design-quality single-page app — beautiful, functional locally, but not a product. Here's the gap.

  • Runs entirely in the browser via Babel standalone.React + JSX is transpiled at page load. Fine for previews, lethal for users — slow first paint, no tree-shaking, no minification.
  • All state lives in localStorage.ps_pantry, ps_saved, ps_grocery, ps_user_recipes, ps_custom_ings, ps_ing_overrides, ps_onboarded — wiping the browser wipes the account. Sharing across devices is impossible.
  • Recipes & ingredients are hard-coded in data.js.Curated, charming, finite. Won't scale past the demo set.
  • No accounts, no auth, no server.No way to recover data, share, sync, or monetize.
  • Fixed viewport width=1280.Desktop-only. Phones get a zoomed-out mess. The whole product concept (pantry while shopping, recipe while cooking) is mobile-first.
  • Zero observability.No error tracking, no analytics, no logs. You won't know if it breaks for someone.
✓
The good news: the visual design system, component architecture, and core interactions (wheel spin, match scoring, cooking mode, onboarding) are solid. You're not rewriting the product — you're rewriting the substrate underneath it.
01
Phase 1Week 1~5 days

Engineering foundation.

Replace the in-browser Babel pipeline with a real build, type the code, and set up the tooling you'll rely on for the next year. Don't add features in this phase — just port what exists.

  • Initialize a Vite + React + TypeScript project.Vite for dev server + bundler; TypeScript for the inevitable refactor pain you'll avoid by typing the data shapes now. pnpm or bun for the package manager.
    Must
  • Port the file structure into src/.Move App.jsx → src/App.tsx, screens into src/screens/, modals into src/components/modals/, shared bits into src/components/, data.js into src/data/. Convert window.X globals to ES module exports.
    Must
  • Define core TypeScript types.Ingredient, PantryItem, Recipe, RecipeStep, GroceryItem, UserPrefs. Type the match-score function, the wheel state, the route enum.
    Must
  • Add ESLint + Prettier + Husky pre-commit.Block commits that don't lint or type-check. Future-you will thank you when refactoring at 11pm.
    Soon
  • Move the CSS into a single source of truth.Lift the design tokens into src/styles/tokens.css (already factored as colors_and_type.css). Per-component styles can stay as plain CSS modules — no need for Tailwind unless you want it.
    Soon
  • Add a real router.Replace the route string-state machine in App.jsx with react-router or tanstack-router. Real URLs (/pantry, /recipes/:id) means link sharing, browser back/forward, and analytics that make sense.
    Must
  • Stand up a state library.zustand for client state (pantry, grocery, UI flags) is plenty — Redux is overkill. Pair with @tanstack/react-query for server state once the API exists.
    Soon
Suggested stack Vite React 18 TypeScript react-router zustand @tanstack/react-query ESLint Prettier pnpm
02
Phase 2Weeks 2–3~8 days

Backend & data layer.

Move state out of localStorage and into a real database behind a real API. This is the single biggest leap toward "production." Pick a backend posture (DIY vs BaaS) and commit.

?
Decision point. For a solo dev shipping fast, Supabase (Postgres + Auth + Storage + RLS) or Convex get you 80% of this phase for free. If you want full control, Node/Hono + Postgres on Fly.io is the more standard path. Pick one before writing schema.
  • Design the relational schema.Roughly the tables below. Use UUIDs as primary keys, created_at / updated_at on every table, soft-delete (deleted_at) on user-content tables.
    Must
📋 Proposed Postgres schema · pantryspin
usersuuid pkemail, display_name, avatar_url, onboarded_at
ingredientsuuid pkname (unique), category, color, swatch, is_global, created_by
pantry_itemsuser_id fkingredient_id, expires_at, location, quantity, added_at
recipesuuid pkname, category, time_min, servings, difficulty, illus_key, is_global
recipe_ingredientsrecipe_id fkingredient_id, qty, unit, is_optional, sub_for
recipe_stepsrecipe_id fkstep_num, body, timer_seconds
recipe_tagsrecipe_id fktag (vegetarian, comfort, healthy, ...)
saved_recipesuser_id fkrecipe_id, saved_at
grocery_itemsuser_id fkingredient_id (nullable), free_text, qty, category, checked_at, source_recipe_id
user_prefsuser_id pkdiet, avoid[], max_time, difficulty, servings, priority
cook_sessionsuser_id fkrecipe_id, started_at, completed_at, rating
  • Decide on global vs per-user content boundary.Global recipes & ingredients are shared, read-only. User-created ones (is_global=false) belong only to the creator. Mirror this exactly in your row-level-security policies.
    Must
  • Build the API surface.REST or tRPC — both are fine. The endpoints map 1:1 to what App.jsx already does: GET /pantry, POST /pantry, DELETE /pantry/:id, GET /recipes, POST /recipes, POST /saved, GET /grocery, PATCH /grocery/:id, GET /me, PATCH /me/prefs.
    Must
  • Write a one-shot migration to seed the current data.js.All the ingredients and recipes you already curated become the initial global content. Don't lose this — it's the soul of the product.
    Must
  • Add input validation at the edge.zod schemas shared client + server. Reject malformed payloads before they hit the DB. Validate every field that comes from the user — especially user-created recipes and ingredient overrides.
    Must
  • Add a local-first sync strategy.The pantry-while-shopping use case needs offline. Keep localStorage as a cache, reconcile on reconnect. react-query + a service worker handles most of this; for true offline-write, look at powersync, RxDB, or just queue mutations.
    Later

03
Phase 3Week 4~3 days

Authentication & accounts.

No accounts = no product. Pick a low-friction auth model and build the smallest possible set of screens around it.

  • Magic-link email + "Sign in with Apple/Google".Skip passwords entirely. Magic links work everywhere, Apple/Google handle 90% of mobile traffic. Supabase Auth or Clerk do this out of the box.
    Must
  • Allow guest mode & later upgrade.The current onboarding doesn't ask for an account — keep that. Create an anonymous user on first launch, store the token, and let them merge into a real account when they sign in. Critical for retention; nobody signs up before they see value.
    Must
  • Migrate the existing localStorage state on first login.Bulk-POST the user's pantry, saved recipes, grocery list, and prefs to the server on first authenticated load. Then clear local. One-way trip.
    Must
  • Account settings screen.Email, display name, avatar, sign out, delete account (with grace period). Required for app store compliance and GDPR.
    Soon
  • Session refresh + multi-device.JWT with refresh tokens, or rely on whatever your auth provider gives you. Test signing in on two devices simultaneously before launch.
    Must
04
Phase 4Weeks 5–6~8 days · ongoing

Real recipe & ingredient content.

The product is only as good as what the wheel can land on. The demo's ~30 recipes won't sustain a returning user past week one. This phase is half engineering, half content ops.

  • Decide where recipes come from.Three viable paths: (a) curate yourself (slow, high-quality, copyrightable), (b) license a feed (Spoonacular, Edamam — both have free tiers but watch ToS), (c) generate with LLMs against a strict schema. Most likely answer: license to bootstrap, curate top picks, let users contribute.
    Must
  • Build an ingredient canonicalization layer."Bell Pepper", "Red Pepper", "Capsicum" should all match. Build an aliases table; use string similarity + manual review for new entries.
    Must
  • Replace the SVG illustrations with real food imagery.The brand SVG thumbnails are charming for a prototype but won't survive next to real apps. Commission a photo set, use a stock service (Pexels / Unsplash with food filters), or generate on-brand illustrations consistently. Store in object storage, serve via CDN with sized variants.
    Soon
  • Build a tiny admin tool for content ops.Even an internal-only Retool / Tooljet board. You'll need to add, edit, retire, and feature recipes weekly.
    Soon
  • Define your minimum recipe surface.200 recipes spread across categories & difficulty is a respectable floor. Anything less and the wheel feels repetitive within a session.
    Soon
!
Legal watch-out. Recipe text is borderline for copyright (ingredient lists alone aren't, but instructions usually are). Don't scrape Bon Appétit. If you license a feed, read the attribution & commercial-use clauses before launch.
05
Phase 5Week 7~5 days

Hardening core features.

The prototype works on the happy path. Production means the long tail of weird states — empty pantries, expired ingredients, abandoned cook sessions, conflicting prefs.

  • Real expiry dates, not "days from now."Currently expiresInDays is a static int. Store an actual expires_at timestamp; compute "expiring in 2 days" at render. Add a daily background job to surface expiring items via notification.
    Must
  • Tune the match-score algorithm.Look at data.js → matchScore. Add weights for expiring ingredients, penalize recipes that need 4+ shop items, factor user prefs (max time, difficulty, diet). Expose ranking signals in an admin debug view.
    Soon
  • Wheel fairness & weighting.Right now the spin picks uniformly from top-4. Weight by match score and freshness so the wheel feels right but never feels rigged. Log every spin → result pair for tuning later.
    Soon
  • Cooking-mode robustness.Wake-lock so the screen doesn't dim. Pause/resume when user backgrounds the tab. Read-aloud option (Web Speech API) is huge for actual cooking.
    Later
  • Empty-state polish & edge cases.Pantry with 0 items shouldn't allow spin. Recipe with all expiring ingredients should get a tag. Grocery list with 0 items has its own onboarding nudge.
    Soon
06
Phase 6Week 8~5 days

Responsive, mobile, PWA, accessibility.

The use case is mobile — at the grocery store, at the stove. The prototype is 1280px-only. Fix this before any real user touches it.

  • Strip the fixed viewport.Change <meta name="viewport" content="width=1280"> to width=device-width, initial-scale=1. Then start the real responsive work.
    Must
  • Mobile shell: bottom tab bar instead of sidebar.5 destinations max — Spin, Pantry, Recipes, Grocery, Me. Sidebar stays for ≥1024px.
    Must
  • Reflow each screen for ~390px wide.Spin: wheel becomes hero, matches scroll horizontally. Pantry: single-column cards. Recipe detail: stack image / meta / ingredients / steps vertically.
    Must
  • Install as PWA.Manifest, icons, service worker. Cache shell + last-seen data. Add-to-home-screen prompt after 2nd session.
    Soon
  • Accessibility pass.Keyboard nav through the wheel, focus rings restored, ARIA on modal close, alt text on illustrations, contrast check on butter-on-cream combos. Test with VoiceOver on iOS.
    Must
  • Touch targets & gesture polish.44px minimum hit area. Swipe-to-delete on pantry chips. Pull-to-refresh on lists. The wheel needs an actual flick-to-spin gesture eventually.
    Soon
07
Phase 7Week 9~4 days · then ongoing

Quality, observability, testing.

If you can't see it break, you can't fix it. Wire up monitoring before users arrive, not after.

  • Error tracking.Sentry, free tier is plenty. Capture unhandled exceptions, source-map your bundles, alert to a Slack/Discord channel.
    Must
  • Product analytics.PostHog (self-host or cloud) or Plausible + Mixpanel. Instrument the 8–10 events that matter: signup, onboarding completed, pantry item added, spin, recipe viewed, recipe saved, cook started, cook completed, grocery added, account deleted.
    Must
  • Server logs & uptime.Whatever your backend host gives you, plus an external pinger (BetterStack, UptimeRobot) on the public health endpoint.
    Must
  • Unit tests on the algorithms.matchScore, the wheel category mapping, the spin-target math. These are pure functions; tests are cheap and catch regressions cold. Vitest.
    Soon
  • End-to-end smoke tests.Playwright covering: complete onboarding → spin → save recipe → start cooking → finish cooking → add to grocery. Run on every PR.
    Soon
  • Performance budget.First contentful paint < 1.5s on a mid-tier Android over 4G. Wheel canvas under 60fps. Lighthouse CI as a PR check.
    Later
08
Phase 8Week 10~3 days

Security, privacy, compliance.

Boring, mandatory, easy to underestimate. Knock it out in one focused week — most of it is configuration, not code.

  • Rate limit the API.Per-user and per-IP. Especially on auth, search, and the LLM-backed endpoints if you add them.
    Must
  • Content Security Policy + standard security headers.CSP, X-Content-Type-Options, Referrer-Policy, Permissions-Policy. Use helmet if Node; most hosts have presets.
    Must
  • Privacy policy, terms of service, cookie banner.Even for a hobby launch — the App Store and EU require both. Use a generator (Termly, iubenda) and have a lawyer eyeball before scale.
    Must
  • GDPR / CCPA basics.Account deletion that actually deletes. Data export endpoint (GET /me/export → JSON). Cookie consent if you ship third-party analytics in the EU.
    Must
  • Secrets management.Nothing in git. Use your host's env-var store. Rotate any keys that ever touched your laptop before launch.
    Must
  • Backups.Daily DB snapshot, 30-day retention, restore drill done at least once before launch. Untested backups don't count.
    Must

Phase 9Weeks 11–12~7 days

Launch infra & growth surfaces.

The product is shippable. Now make sure it can be deployed, marketed, supported, and grown — without you babysitting it 24/7.

  • CI/CD with preview deploys.GitHub Actions or Vercel/Netlify built-in. Every PR gets a unique URL. main deploys to staging; tagged releases promote to prod. Manual approval on prod for the first month.
    Must
  • Custom domain + TLS + email.pantryspin.app or similar. Set up transactional email via Resend or Postmark — you'll need it for magic links, weekly digests, expiring-pantry alerts.
    Must
  • Marketing landing page.Separate route or subdomain. Three sections: hero with wheel demo, the three core jobs (reduce waste, decide faster, shop less), email capture. Keep it shorter than this document.
    Soon
  • Push notifications & weekly digest.Web Push for "you have 2 items expiring tomorrow." Weekly email summarizing what you cooked + suggesting 3 recipes from your pantry.
    Later
  • Native shell — Capacitor wrap.When you're ready for iOS/Android, wrap the PWA in Capacitor for App Store distribution. Reuse 95% of the codebase. Apple needs an explicit account-deletion path; Android needs a privacy form.
    Later
  • Support & feedback loop.In-app "Send feedback" button → routes to email or a Linear/Tally form. Public changelog. Status page if you reach >1k DAU.
    Soon
→
Closed beta first. Don't skip this. Invite 25–50 people, give them a Discord channel, fix their top complaints, then open up. Public launches are very hard to redo.

DecisionsResolve before week 1

Decisions still to make.

Nine open questions whose answers shape every phase above. Don't start Phase 1 with these unresolved — the cost of changing them later compounds.

Before you write any code

Answers to these determine your stack, your timeline, your hosting bill, and your TOS.

01
Backend posture — BaaS or DIY? Supabase/Convex saves 2 weeks but locks the data model into their conventions. DIY keeps options open at the cost of velocity. Pick one this week.
02
Free, freemium, or paid? Free forever (ads / future tier), free with a paid Pro ($3–5/mo for things like meal planning + family share), or paid upfront. Affects whether Stripe is in scope.
03
Recipes — license, generate, or curate? Cheapest answer is license a feed to bootstrap; most defensible is curated + LLM-assisted. Decide before phase 4.
04
Native or PWA only? PWA covers 90% of needs at 10% of the cost. Apple's PWA support is improving but still second-class. If you need notifications on iOS today, Capacitor wrapper is required.
05
Multi-user households? Sharing a pantry across roommates / a family changes the data model significantly. Cheaper to design for it now than retrofit.
06
Barcode scanning & receipt OCR? Highest-value feature for the pantry-tracking job, also the most expensive. v1 or v2?
07
AI features in-scope for v1? "Spin again but vegetarian," "what can I make in 15 minutes," recipe-from-photo. These ride on the wheel concept beautifully but cost $$ per call.
08
Where do you host? Vercel + Supabase is the path of least resistance. Cloudflare Workers + D1 is cheaper at scale. Fly.io + Postgres if you want one stack to learn.
09
What's the launch metric? "1,000 weekly active users with >3 spins/week" is a real bar. "Got it on the App Store" is not. Pick something measurable so phase 7 instruments toward it.
